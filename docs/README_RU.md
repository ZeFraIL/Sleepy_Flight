🎯 Цель проекта
• **Какую задачу решает приложение:** Обеспечивает воспроизведение белого шума для сна во время полета, но только при соблюдении условий безопасности (включен «Режим полета») и вежливости (подключены наушники).
• **Почему эта задача важна:** Это предотвращает случайное включение громкого звука в салоне самолета и напоминает пользователю включить авиарежим для экономии батареи и безопасности.
• **Целевая аудитория:** Часто летающие пассажиры, путешественники и люди, использующие звуки природы или белый шум для сна.
________________________________________
📌 Требования к приложению
**Функциональные требования**
• Определение статуса режима полета.
• Определение подключения наушников.
• Управление воспроизведением (Play/Pause) на основе условий и действий пользователя.
• Обновление интерфейса при изменении состояния системы.
**Нефункциональные требования**
• **Производительность:** Минимальное потребление энергии в фоновом режиме.
• **Удобство использования:** Простой интерфейс на одном экране.
• **Надёжность:** Точное отслеживание системных событий через BroadcastReceiver.
________________________________________
🧠 Общая архитектура
• **Выбранный подход:**
– MVC (Model-View-Controller)
• **Почему выбран именно он:** Проект компактный. Использование MVVM было бы избыточным для приложения с одним экраном и минимальной логикой данных.
• **Основные компоненты системы:**
– `MainActivity` (Вид и Контроллер)
– `AirplaneReceiver` (Обработчик системных событий)
– `HeadphonesReceiver` (Обработчик системных событий)
– `MediaPlayer` (Ресурс/Модель)

🧩 Подробное описание классов
📌 **Класс: MainActivity**
**Роль:**
Центральный узел приложения.
**Ответственность:**
Управление интерфейсом, инициализация MediaPlayer и координация логики при изменении состояний.
**Основные методы:**
- `onCreate()` — Инициализирует компоненты UI и регистрирует ресиверы.
- `checkConditionsAndPlay()` — Проверяет, включен ли авиарежим и подключены ли наушники перед запуском звука.
- `updateUI()` — Обновляет текстовые индикаторы на экране.
- `onDestroy()` — Освобождает ресурсы медиаплеера.
**Взаимодействие с другими классами:**
Получает обновления от `AirplaneReceiver` и `HeadphonesReceiver`.
________________________________________
📌 **Класс: AirplaneReceiver**
**Роль:**
Слушатель системных событий.
**Зачем используется:**
Чтобы автоматически реагировать на переключение режима полета без необходимости постоянного опроса системы приложением.
________________________________________
📌 **Класс: HeadphonesReceiver**
**Роль:**
Слушатель системных событий.
**Зачем используется:**
Гарантирует, что звук будет играть только в наушниках, не мешая окружающим.
________________________________________
🔄 Диаграмма работы приложения
1. Пользователь открывает приложение.
2. Приложение проверяет текущий статус авиарежима и наушников.
3. Пользователь нажимает "Play".
4. Если (Авиарежим=ВКЛ И Наушники=ПОДКЛЮЧЕНЫ), начинается воспроизведение.
5. Если во время игры пользователь выключает авиарежим, звук мгновенно прекращается.

🎨 UI/UX анализ
• **Почему интерфейс сделан именно так:** Акцент на ясности. В темном салоне самолета пользователь должен мгновенно видеть статус.
• **Какие принципы использованы:**
– **Простота:** Одна кнопка, две строки статуса.
– **Логичность:** Понятная текстовая обратная связь.
– **Доступность:** Крупный текст и кнопки.
• **Что можно улучшить:** Добавить таймер сна или выбор различных звуков.
________________________________________
⚙️ Работа с потоками
• **Используются:**
– Внутренние потоки `MediaPlayer` (асинхронная подготовка и воспроизведение).
• **Почему выбран этот способ:** Стандартный `MediaPlayer` Android эффективно выносит декодирование аудио за пределы основного потока.
• **Как предотвращаются:**
– **Зависания (ANR):** Отсутствие блокирующих операций ввода-вывода в основном потоке.
– **Утечки памяти:** В методе `onDestroy` вызывается `release()` для медиаплеера.
________________________________________
💾 Работа с данными
• **Где хранятся данные:** Постоянное хранилище не используется (приложение без сохранения состояния).
• **Почему выбран этот способ:** Поведение приложения полностью зависит от текущих состояний системы в реальном времени.
________________________________________
🌐 Работа с сетью
• **Статус:** Сеть не используется. Это ключевая особенность, так как приложение предназначено для работы в автономном режиме полета.
________________________________________
🔐 Безопасность (базовый уровень)
• **Чувствительные данные:** Отсутствуют.
• **Защита:** Не требуются специальные разрешения, кроме стандартных слушателей состояния системы.
